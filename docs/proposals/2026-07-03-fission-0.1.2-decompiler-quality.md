# Fission 0.1.2 Decompiler Quality Proposal

This proposal applies ADR 0006 before changing production decompiler semantics.
The goal is to reduce compile/runtime blockers without overfitting to a
function name, address, binary id, corpus row, or compiler-specific label.

## Slice 1: Compile Blockers

### 1. Baseline Row Anchor

- Binary: `corpus/dev/binaries/control_flow_gcc_O0.exe`
- Function: `checksum`
- Address: `0x1400015b4`
- Corpus row or benchmark command: `fission-benchmark` dev corpus, Fission row
- Current output summary: scalar accumulator/local counter materializes as
  `uchar *`, producing `local_4 %= 256`.
- Semantic cases passed / total: `0 / 5`
- Failure category: `compile_error`
- Relevant observations: same role-confusion family appears in `crc32` as
  pointer XOR and in `rc4_*` as scalar arithmetic on pointer-typed temporaries.

- Binary: `corpus/dev/binaries/crypto_gcc_O0.exe`
- Functions: `rc4_init`, `rc4_crypt`, `crc32`
- Addresses: `0x140001530`, `0x140001624`, `0x140001745`

- Semantic cases passed / total: `0 / 5`, `0 / 4`, `0 / 6`

Evidence:

```text

uchar * local_4;

```

The first wrong facts are role/type facts attached to materialized locals and
parameters before the final printer renders declarations.

### 3. Generality / Invariant Proof

Generalized rule:

```text
When one storage location is reused for incompatible roles, the emitted C
binding must not preserve a stale pointer role for scalar-only arithmetic,
bitwise, shift, modulo, compare, or loop-index uses. Pointer role should be kept
only for address-producing uses such as load/store base expressions or typed
pointer parameters.
```

Comparable coverage:

- Similar shape 1: `checksum` byte-load pointer plus scalar sum/index locals.
- Similar shape 2: `crc32` byte-load pointer plus scalar CRC accumulator.
- Synthetic invariant test: a local assigned from a pointer load address and
  later used as `%`, `^`, `&`, `>>`, or scalar loop counter must be declared as
  an integer scalar unless its remaining uses are address-producing.

### 4. Risk And Ownership Check

- Existing owner: materialize binding selection and type/data recovery.
- Extend existing owner first; do not add a new pass unless the invariant cannot
  be represented in current type-role selection.
- Risk: over-correcting pointer parameters into integers. Parameters used as
  load/store bases must retain pointer declarations.
- Risk: hiding useful flag semantics. Flag temporaries should either be declared
  consistently or eliminated only when proven dead/non-semantic.

## Slice 2: `count_bits` Runtime / Parity

- Binary: `corpus/dev/binaries/control_flow_gcc_O0.exe`
- Function: `count_bits`
- Address: `0x140001530`
- Current output summary: loop body computes `uVar13 = param_10 & 1`, accumulates
  into `local_4`, but returns `uVar13`.
- Semantic cases passed / total: `2 / 6`
- Failure category: `runtime_error`

### 2. Owner Proof

- [ ] SLEIGH/raw p-code:
- [x] Builder/materialize:
- [x] Return-value recovery:
- [ ] Normalize:
- [ ] Structuring:
- [ ] Printer:
- [ ] Benchmark/automation:

Evidence:

```text
local_4 += uVar13;
param_10 >>= 1;
...
return uVar13;
```

The wrong fact is the selected return source after a loop-carried accumulator,
not a dashboard or wrapper issue.

### 3. Generality / Invariant Proof

Generalized rule:

```text
For a loop with a scalar accumulator updated on the backedge and a final return
register load from that accumulator after loop exit, return recovery must select
the accumulator value reaching the exit, not the last temporary used to update it.
```

Comparable coverage:

- Similar shape 1: O0 `count_bits`, stack local accumulator plus bit temp.
- Similar shape 2: O2 `count_bits`, register accumulator plus loop induction
  value.
- Synthetic invariant test: loop-carried accumulator update followed by a final
  return load returns the accumulator binding.

### 4. Risk And Ownership Check

- Existing owner: materialize loop-carried merge binding and terminator return
  recovery.
- Do not broaden CFG structuring first. Use CFG/postdom facts only if they prove
  the loop exit return source.
- Risk: changing unrelated return selection for functions with calls or stores
  near the exit.

- [ ] Targeted invariant test:

  - Expected signal: return expression uses the accumulator binding.
- [ ] Crate-level gate:

  - Expected signal: all non-ignored tests pass.
- [ ] Focused benchmark row:

  - Expected row-level improvement: `count_bits` semantic score improves and no
    compile blocker regresses.
  - Local verifier result after this slice: `count_bits` improved to `6 / 6`
    semantic cases passed.
