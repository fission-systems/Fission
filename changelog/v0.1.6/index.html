    Fission v0.1.6¶

    Fission 0.1.6 is a Ghidra-parity and correctness release. Where 0.1.5 focused on determinism and speed, 0.1.6 closes a comprehensive audit of debug-info extraction quality (DWARF and PDB), lands whole new Ghidra-analyzer-equivalent features (FID library-function identification, Windows TEB/PEB recognition, AggressiveInstructionFinder-style gap scanning, X86FunctionPurgeAnalyzer, FormatStringAnalyzer), and fixes a run of real decompiler-correctness bugs — several of them causing either a hard crash or a silently wrong decompiled output (a dropped write, a wrong return value, a wrong type) rather than a merely-suboptimal one. Every fix in this release was validated against a real compiled fixture (mingw/gcc, and where relevant a real Ghidra 12.0.4 database comparison) before landing, not synthetic-only.

    This remains a pre-1.0 minor release: the documented CLI is the primary interface. NIR remains the semantic oracle; HIR is a clone-only readability polish layer.

    Tagged commit: d2ee0e32 Release: https://github.com/sjkim1127/Fission/releases/tag/v0.1.6

    DWARF/PDB debug-info quality¶

    A systematic audit (prompted by asking whether debug-info metadata Fission already collects is fully consumed) found six real gaps, all closed this release:

    • Register-resident DWARF locals. Previously only stack-offset DWARF variables got real names; anything a compiler kept in a register at -O1+ stayed generic (uVar0). Fixed by wiring up .debug_loc/.debug_loclists (previously hardcoded to empty sections) and Ghidra's own per-architecture DWARF-register-number tables, with a location-list-agreement check that only renames a binding when every range agrees on the same register. Verified against real GCC 16 -O1 output. Two unrelated, pre-existing DWARF bugs surfaced only because this was the first real GCC (not clang/zig-cc) end-to-end test and were fixed along the way: DW_AT_high_pc read as a raw absolute address without checking the offset-from-low_pc form GCC actually uses (collapsed affected functions' sizes to 0), and an off-by-one DFS-depth check that silently folded a function's name/params/locals into the previous function's entry.
    • DWARF enum values, not just enum type names — DW_AT_const_value extraction per enumerator, including negative-value support.
    • DWARF array type names — previously always resolved to the literal string "unknown"; now resolves multi-dimensional names like int[3][4].
    • PDB struct layouts and variable locations — Windows PDB had none of the struct/local-location extraction DWARF already had. Validated against a real Windows PDB: 102 struct types extracted, and 80 of 154 (52%) of the binary's parameters gained a real stack location where every one had previously been Unknown.
    • .debug_line parsing — the line-number program was section-loaded but never executed; now flattened into an address-sorted, binary-searchable table, cross-validated byte-for-byte against llvm-dwarfdump. Extraction only in this release — no rendering consumer wired yet, deliberately kept as a separate follow-up.
    • DW_TAG_lexical_block PC-range scoping — nested-block-scoped locals (e.g. three C blocks each shadowing an outer variable) previously merged into indistinguishable flat entries; now scoped to each variable's innermost enclosing block's real PC range.

    C++ exception handling and RTTI¶

    • GCC/Itanium LSDA (.gcc_except_table), implemented by hand-decoding real g++ try/catch fixtures (including a PIE build needing GOT-indirect type-table resolution) against readelf/objdump before writing any code. Initially extraction-only; a follow-up found a catch block was silently absent (not just unstructured) from decompiled output because its landing pad wasn't seeded into decode/reachability at three independently-broken pipeline layers — all three fixed. Two more follow-ups fixed generic "no textual reference = dead label" cleanup passes deleting a catch handler's own label (one in structuring, one covering six instances of the same pattern in normalize). A real catch block now renders as actual code for the first time.
    • Windows .pdata/.xdata SEH (mingw-w64 g++) — cross-compiling a real try/catch fixture found mingw's C++ personality routine emits the identical GCC/Itanium LSDA byte format, just relocated into .xdata's handler-data area instead of .eh_frame — the ELF parser was refactored into a shared, producer-agnostic core and reused for PE with zero further downstream changes needed.
    • Itanium __vmi_class_type_info (multi/virtual-inheritance RTTI), validated byte-for-byte against a real g++ fixture. Along the way, found the entire existing Itanium RTTI analyzer was non-functional on any real ELF binary — symbol matching used raw mangled names that never matched Fission's already-demangled table, and the discriminator broke on dynamically-linked binaries with an external vtable. Both fixed.
    • Split DWARF (.gnu_debuglink/.note.gnu.build-id) — the default packaging for most distro system libraries (Debian -dbgsym, Fedora debuginfo) previously produced zero DWARF data for Fission. Both resolution conventions implemented with CRC32 staleness validation, verified against a real split-debug fixture built the standard distro way.
    • Call-Fixup mechanism, investigated separately — the real bugs turned out to be two stack-address-resolution gaps (a non-literal sub rsp, rax delta, and rbp established at a nonzero frame offset misclassifying locals as parameters), not Ghidra-style call-stub substitution as originally assumed. Both fixed and confirmed to silently correct two functions already in the golden-corpus snapshot. 32-bit __x86.get_pc_thunk.* remains open.

    FID (Function ID) library-function identification¶

    Found completely dead despite a complete .fidbf database parser already existing: nothing computed a query hash, so statically-linked library functions (memcpy, OpenSSL, …) in stripped binaries were never identified. Implemented end-to-end, validating each step against real Ghidra 12.0.4 output before trusting it: instruction pattern masks, the FNV-1a full hash, simple-memory-operand and SIB (base+index*scale) addressing, RIP-relative addressing (found already correct by construction), and the "specific hash" (+10 score bonus). Wired into fission_cli as a standalone identify subcommand. A separate, pre-existing, structurally-broken FID hashing pipeline already silently wired into decompiler call-target renaming (wrong FNV variant, register operands never captured) was found and replaced. AArch64 validation then found stp/ldp register-pair addressing — present in nearly every non-leaf function's prologue/epilogue — fell through unhandled, meaning almost no AArch64 function could be hashed at all; fixed, along with an x86-only NOP-skip pattern list that was being applied unconditionally to every architecture. First live matches (independently confirmed against the binaries' own unstripped symbol tables): __register_frame_table (libgcc, x86-64) and _Unwind_GetIPInfo/__deregister_frame (libgcc, AArch64).

    Windows-specific Ghidra parity¶

    A full survey of Ghidra's 54 non-scope-excluded analyzer classes (24 already covered, 12 legitimately out of scope, 18 real gaps found across two rounds) drove this section:

    • Classic 32-bit __chkstk/__alloca_probe, per Ghidra's own x86win.cspec (ESP = ESP + 4 - EAX) — distinct from mingw's ___chkstk_ms, already correctly handled. Not validated against real MSVC-produced bytes (no MSVC toolchain available in this environment).
    • TEBAnalyzer — Windows TEB/PEB fs:/gs:-relative field access, including the full PEB.BeingDebugged anti-debug check chain (fs:/gs: → TEB.ProcessEnvironmentBlock → +0x2). Validated against real 32- and 64-bit mingw fixtures.
    • SharedReturnAnalyzer — audited and found Fission already has an equivalent function-boundary/call-graph mechanism; the real gap was in rendering: a direct-address tail call (jmp known_func) rendered as a bare statement instead of return known_func(), silently implying execution continued when it didn't. Confirmed on a real mingw CRT function (__gcc_register_frame tail-jumping into atexit), already present in the golden corpus.
    • AggressiveInstructionFinderAnalyzer — self-referential prologue fingerprinting (a candidate's instruction shape must recur ≥4 times among the binary's own already-known functions) to recover function starts hidden in stripped-binary gaps, without a hardcoded signature database. Gated to the Aggressive discovery profile only (previously a no-op relative to Balanced). Found and fixed an unrelated, 100%-reproducible crash in an adjacent scanner (AhoCorasick built with the wrong match kind for the search mode it was used with) while testing this.
    • X86FunctionPurgeAnalyzer — recovers a callee-cleanup (stdcall/ fastcall) function's exact stack-argument count from its own RET imm16 purge byte, forcing unused trailing parameters back into the recovered signature instead of silently dropping them. Validated against a real 32-bit mingw fixture with a genuinely unused trailing parameter.
    • FormatStringAnalyzer — types printf-family variadic arguments from their own format string's %-specifiers (printf("%d %s", x, y) now types x as int, y as char*). Found and fixed two real, pre-existing limitations in the shared type-propagation machinery along the way (a type-tightening helper that couldn't override the generic register-width default every real call argument starts with — confirmed the existing WinAPI-signature typing had the exact same silent limitation — and a copy-chain-tracing gap that dropped a correctly-computed type before it reached the real source parameter). Deliberately scoped to the unambiguous ANSI narrow-string printf family; scanf-family and wide-character wprintf-family are intentionally not covered.

    Decompiler correctness fixes¶

    • 8 missing FLOAT_* p-code opcode lowerings, fixing a hard crash. FloatFloat2Float, FloatNeg, FloatAbs, FloatSqrt, FloatCeil, FloatFloor, FloatRound, and FloatTrunc had zero lowering handlers, hitting a generic "unsupported pcode pattern" error that failed decompiling the entire function — found via a real 32-bit x87 floating-point fixture (i686-w64-mingw32-gcc). None of these opcodes are x87-specific at the p-code level, so any code using them (x87 or otherwise) hit the identical crash. FloatTrunc specifically is a float-to-int truncating conversion, not a call to trunc() — verified against Ghidra's own TypeOpFloatTrunc C++ declaration (TYPE_INT, TYPE_FLOAT) rather than assumed from the opcode's name.
    • A subvariable-bit-width-narrowing pass could fabricate a phantom, uninitialized-looking local declaration for any def-less variable name it encountered mid-trace, including names that were never real storage in the first place (surfaced by the new TEBAnalyzer/FormatStringAnalyzer work, which both deliberately use unregistered synthetic names). Fixed at the root — the pass's leaf case now requires the name be a genuinely declared local/parameter — not worked around in the newer features that happened to surface it.
    • Dead-store elimination could silently drop a real write through a pointer value spilled to an ordinary stack slot (the shape a variable-length array's dynamically-computed base address takes once the compiler spills it) — found via a real fixture (int arr[n], genuinely dynamic n) where arr[i] = ... disappeared from the decompiled output entirely, at both the NIR and HIR layers. A same-shaped fixed-size array (int arr[16]) was unaffected, confirming this was VLA-specific, not a general array-write regression.

    Known issues (not fixed this release)¶

    Found during this release's audits, characterized with real fixtures, but scoped larger than a single-release slice — recorded here rather than left silent:

    • x87 FPU register-stack reconciliation. SLEIGH lifts FLD/FSTP as an explicit 8-fixed-register Copy-rotation modeling the hardware stack; Fission's materialize/copy-propagation pipeline doesn't specially recognize this idiom. A genuinely double-precision x87 function now decompiles (the crash above is fixed) but with confusing st0/st6/st7-named temporaries and, in the observed real fixture, an incorrect final return value (x87's actual return convention — the value left on ST0 — isn't recognized by return-value recovery, which is built around general-purpose "primary return registers" only).
    • 32-bit stack-passed double/int64 parameters can be misread as two separate 4-byte int parameters when the compiler copies them through a realignment scratch buffer before an fldl (observed on a real mingw 32-bit fixture) — a different, likely more common-impact issue than the x87 return-value one, not yet investigated to a fix.
    • 32-bit __x86.get_pc_thunk.* call-fixup remains unhandled (noted above).

    Upgrade notes¶

    • No intentional CLI contract break relative to 0.1.5.
    • Decompiled output for functions that hit any of the fixes above (x87 float-op-using functions, VLA-indexed functions, direct-address tail calls, printf-family variadic call sites, 32-bit stdcall functions with unused trailing parameters) will differ from 0.1.5 — in every case, this release's output is the correct one. The golden corpus itself changed for exactly one function (__gcc_register_frame, present in all 16 corpus binaries via the shared mingw CRT) due to the tail-call return fix, reviewed and confirmed intentional before accepting the new snapshot.
    • New fission_cli identify subcommand for standalone FID-based library-function identification.
    • --function-discovery-profile aggressive now differs from balanced for the first time (previously an accidental no-op) — the self-referential prologue-fingerprint scanner only runs under aggressive, matching Ghidra's own AIF being strictly riskier than the reference/signature- driven scanners that run under balanced too.
    • fission-benchmark operators: set FISSION_VERSION=v0.1.6 (release source only for official/Pages).
