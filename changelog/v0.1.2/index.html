    Fission v0.1.2¶

    Fission 0.1.2 is a decompiler-quality, SLEIGH parity, and execution-foundation release. Compared with 0.1.1, it substantially expands semantic recovery in the NIR pipeline, tightens raw p-code agreement with Ghidra, introduces dual NIR/HIR rendering, and adds experimental native emulation and solver foundations.

    This is a large patch release during Fission's pre-1.0 development period. The documented CLI remains the primary supported interface, but output produced by the decompiler can change as recovery and structuring improve. The new sandbox, symbolic execution, and native solver surfaces are experimental.

    Release summary¶

    Area v0.1.1 v0.1.2
    Decompiler feedback DecompContext and fact feedback foundation Broader variable, type, return, loop, and conditional recovery built on that foundation
    P-code parity Initial native SLEIGH pipeline Ghidra-aligned branch targets, inst_next, USEROP indexing, and conversion semantics
    Output layers Single primary pseudocode path Explicit NIR/HIR presentation selection with decomp --layer
    CFG structuring Iterative structuring feedback Additional diamond, natural-loop, epilogue-return, short-circuit, and tail-CMOV recovery
    Type recovery Initial fact propagation Storage-role separation, reaching definitions, partial-register handling, and scalar/address constraints
    Execution Debugger-oriented dynamic layer Experimental JIT-only user-mode p-code sandbox with Linux/Windows HLE
    Constraint solving No native solver crate Experimental AIG/CNF, CDCL SAT, bit-vector, array, and path-feasibility foundations
    Change governance Pass-gate architecture ADR-backed anti-overfit, ISA-neutral ownership, validation-pool, and audit policies
    CI posture Standard lint/test/release lanes Exact parity and zero-tolerance regression gates, changed-file formatting ratchet, stronger security policy
    Package version Release tag v0.1.1; workspace packages still reported 0.1.0 Release tag, all workspace packages, and fission_cli --version report 0.1.2

    The implementation range contains 98 non-merge commits and, before the release metadata update, changes 315 files with approximately 41,110 insertions and 2,113 deletions. Much of that growth is isolated in the new experimental emulator and solver crates rather than the core decompiler path.

    Decompiler semantic recovery¶

    Variable identity and storage roles¶

    0.1.2 makes variable recovery less dependent on a single physical register or stack storage identity. Important changes include:

    • preventing variable merge from collapsing simultaneously live general-purpose registers;
    • splitting incompatible storage roles across program phases, such as a pointer cursor, loaded byte, loop index, and scalar accumulator sharing one machine location over time;
    • tracking address contributors through redefinitions instead of losing pointer evidence at the most recent assignment;
    • preserving transitive address-local widths and exact scalar-local constraints;
    • improving RC4-style byte-state recovery and unsigned low-byte truncation;
    • recovering RC4 byte-sum indices formed through zero extension and arithmetic;
    • keeping primary return materializations when the ABI return register remains live at function exit; and
    • narrowing signed 32-bit return values recovered through join temporaries and select expressions.

    These are general data-flow and type constraints. Production behavior does not branch on benchmark function names, binary identities, or fixed addresses.

    Returns, conditions, and loops¶

    The structuring and materialization pipeline now handles more common optimized and unoptimized control-flow shapes:

    • clamp-like conditional returns whose comparisons previously disappeared;
    • x86-32 diamond returns joined through a shared epilogue;
    • natural-loop returns and loop-carried self-updates;
    • x86-32 conditional-move clamp forms;
    • short-circuit diamonds whose return-value arm must remain materialized;
    • tail-of-block absolute conditional-move skips;
    • return-bearing segments in if/goto fusion;
    • epilogue joins where the external condition-code seed is not a valid ownership barrier; and
    • signed INT_MIN presentation and partial-register xor plus setcc composition.

    The work is deliberately shape-based. It uses CFG, dominance, liveness, definition/use, ABI, and type facts rather than recognizing named corpus rows.

    x86 recovery details¶

    • x86 register-space EFLAGS bits receive stable names for Jcc recovery.
    • dead EFLAGS assignments are removed when they cannot affect observable output.
    • x86-32 stack parameters are recovered from calling-convention and stack facts.
    • logical right-shift parameters are typed as unsigned only in the logical-shift context.
    • partial-register zero extension and low-byte masks feed type recovery without forcing a whole-register identity merge.

    SLEIGH and Ghidra p-code parity¶

    0.1.2 includes a focused parity slice for Fission's native SLEIGH runtime.

    • inst_next is resolved with a two-pass decode/lift strategy so RIP-relative forms with trailing immediates use the correct next-instruction address.
    • Relative p-code BRANCH targets are interpreted relative to the current p-code operation where required.
    • USEROP indices are dense and stable across compiled specifications.
    • Size-aware flag operations preserve operand-width semantics.
    • Floating-point conversion operation names and lowering are aligned with the shared p-code contract used by parity tooling.
    • Runtime decode, template walking, function lifting, and raw p-code serialization were updated together so CLI telemetry and internal execution see the same IR.

    Ghidra is used as a structural reference for assembly, p-code, CFG, and function inventory diagnostics. Fission does not attempt to reproduce Ghidra's C syntax or presentation quirks, and parity metrics are not semantic-correctness scores.

    NIR and HIR presentation¶

    The CLI now exposes explicit decompiler presentation layers:

    fission_cli decomp program.exe --addr 0x140001000 --layer nir
fission_cli decomp program.exe --addr 0x140001000 --layer hir
    • NIR presentation keeps lower-level storage and data-flow evidence useful for decompiler diagnosis.
    • HIR presentation applies higher-level rendering intended for human inspection.
    • JSON output records the selected layer so automation does not need to infer it from formatting.
    • Printer changes include improved signed constants, return expressions, type spelling, and structured control-flow rendering.

    Consumers that compare pseudocode text byte-for-byte should expect snapshot updates. Prefer JSON fields and semantic tests for stable automation.

    Experimental native sandbox and emulator¶

    0.1.2 introduces fission-emulator and connects it to the CLI through the experimental sandbox command. This work is an execution foundation, not a claim of general binary compatibility.

    Implemented foundations include:

    • a JIT-only p-code execution path based on Cranelift;
    • page-based memory, register-space mapping, loader integration, and execution budgets;
    • PE mapping and selected Windows x64 HLE procedures;
    • ELF mapping, selected Linux syscalls, dynamic-linker scaffolding, relocations, and selected libc HLE procedures;
    • architecture and calling-convention descriptions separated from OS behavior;
    • copy-on-write symbolic memory and taint propagation;
    • snapshots, time-travel state integration, and Semantic Replay Diff telemetry;
    • exploration techniques, a SimulationManager-style state set, path-feasibility pruning, and deterministic instruction limits; and
    • structured metrics for unimplemented operations, exits, HLE calls, and budget exhaustion.

    The included ELF and PE fixtures are small smoke and diagnostic programs. Real programs can still stop on unsupported p-code, syscalls, relocations, floating point behavior, dynamic linking, or OS APIs. Sandbox output and command-line options may change in subsequent 0.x releases.

    Experimental native solver¶

    The new fission-solver crate provides foundations used by symbolic execution:

    • expression AST simplification;
    • AIG construction and bit blasting;
    • CNF generation;
    • CDCL SAT solving with watched propagation, VSIDS, phase saving, assumptions, and model extraction;
    • initial bit-vector and array-theory support; and
    • incremental path queries and CEGAR-oriented memory reasoning.

    This is not yet a drop-in replacement for mature SMT solvers. Theory coverage, floating-point semantics, diagnostics, and performance remain under active development. The APIs are experimental and are not covered by the stable CLI contract.

    Loader and dynamic-analysis changes¶

    • ELF loading now exposes additional mapping and dynamic-linking information needed by the emulator.
    • PE/COFF loading exposes imports and image facts used by Windows HLE.
    • Function views preserve more precise boundaries for downstream static and emulation consumers.
    • The dynamic debugger can select an emulator backend while retaining the platform debugger abstraction.
    • TTD snapshots can carry additional execution state required for deterministic replay.

    Architecture and anti-overfit guardrails¶

    0.1.2 adds process and module-level controls intended to keep decompiler fixes general:

    • ADR 0006 requires an owner proof, invariant proof, and validation matrix before semantic quality changes.
    • ADR 0007 prevents benchmark identifiers from entering AI review prompts and defines an unseen patch-validation pool.
    • ADR 0008 narrows fission-pcode toward an IR and semantic substrate, with explicit analysis, normalization, and structuring ownership boundaries.
    • ADR 0009 requires ISA-neutral semantic rules unless architecture-specific facts are supplied through the architecture abstraction.
    • SKILL.md packages the decompiler improvement workflow for use in other AI development environments.
    • Audit scripts scan for benchmark smells, prompt leaks, architecture-isolation violations, NIR boundary violations, regression risk, and metric gaming.
    • Gate policy is versioned in TOML rather than hidden in CI conditionals.

    These controls do not prove that every recovery rule is universally correct. They make the evidence and ownership required for each rule reviewable.

    Automation, CI, and security¶

    • CI now enforces zero-tolerance regression and exact-match parity lanes where the corresponding contract is designated exact.
    • Changed Rust files are checked by a formatting ratchet without requiring an unrelated repository-wide rewrite.
    • Automation quality policy is separated from report rendering and loaded from a versioned configuration file.
    • Fast-gate jobs cover lint, security, architecture policy, and targeted test surfaces before release tagging is allowed.
    • Cargo advisory policy was updated for the Cranelift/memmap dependency context; the exception is narrowly scoped and documented rather than globally ignored.
    • Release tags are still created only by the Release Tag (CI green) workflow after the selected commit has a successful Fast Gate run.

    CLI surface¶

    The release CLI includes the following top-level command families:

    info, list, disasm, raw-pcode, pcode-stages, nir-stats,
pcode-topology, decomp, strings, xrefs, callgraph, inventory,
resources, script, debug, sandbox, ai

    sandbox and its symbolic/exploration features are experimental. Existing metadata, listing, disassembly, p-code, decompilation, resource, and automation commands remain the primary release surface.

    Compatibility and migration notes¶

    • The executable name remains fission_cli.
    • Release archives continue to bundle required signatures, Ghidra-derived data, and SLEIGH resources under the existing layout.
    • fission_cli --version now reports 0.1.2; 0.1.1 archives could report the older workspace package version despite the release tag.
    • Decompiled pseudocode is expected to differ because variable identity, types, returns, and control flow have changed.
    • Tools parsing human-readable pseudocode should migrate to JSON output where a machine-readable mode exists.
    • The new NIR/HIR layer selector should be set explicitly in snapshot-based automation.
    • No compatibility guarantee is made yet for Rust APIs in fission-emulator or fission-solver.

    Validation evidence¶

    The release candidate is validated through the repository release gate, which includes:

    • workspace metadata and locked dependency resolution;
    • cargo check for the release CLI and affected semantic crates;
    • the fission-pcode nextest suite;
    • automation policy tests;
    • lint, formatting, architecture-policy, and security checks; and
    • cross-platform CLI builds performed by tag-driven CD for Linux x86-64, macOS arm64, and Windows x86-64.

    The CD workflow also runs fission_cli --version, packages resource data, produces SHA-256 checksums, and performs a Linux benchmark-consumer smoke check.

    Benchmark interpretation¶

    The last checked 0.1.1 benchmark envelope is a diagnostic baseline, not evidence of 0.1.2 improvement. Its layered Fission-to-Ghidra parity report recorded:

    • assembly comparable-row match: 79.2% (95/120), 100% coverage;
    • p-code opcode-sequence agreement: 96.7%;
    • p-code loose and strict full-row agreement: 66.7%;
    • CFG block-start Jaccard: 98.8% and edge Jaccard: 96.8%, while exact CFG set agreement remained 7.8%; and
    • function-inventory presence recall: 94.6%, with exact set agreement at 50.0%.

    Those values identify the remaining work; they are not a multi-decompiler quality ranking and do not establish semantic equivalence. The benchmark repository is pinned to v0.1.2 after release assets are published, then rerun so 0.1.2 receives its own provenance-linked measurements.

    Known limitations¶

    • Decompiler semantic correctness remains incomplete, particularly on complex aliasing, optimized data flow, indirect control flow, and broad multi-ISA corpora.
    • Exact CFG and function-inventory parity with Ghidra is not yet reached.
    • Raw p-code full equality can differ even when opcode sequences agree because varnodes, address spaces, and function-discovery policies remain stricter.
    • The emulator supports selected user-mode paths and HLE procedures, not a full operating system or complete ISA/OS matrix.
    • The solver has incomplete theory coverage and is intended for controlled path queries rather than unrestricted SMT workloads.
    • Readability proxy metrics remain unvalidated until the planned human comprehension study; they are not a release-quality score.
    • Benchmark semantic results remain separate from structural parity and source similarity.
